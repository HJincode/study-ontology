# Ontology 기본 개념 예제 (초급 / 중급 / 고급)

온톨로지(Ontology) 기본 개념을 단계별로 익히기 위한 예제 프로젝트의 **기본 틀**입니다.  
코드는 포함하지 않았으며, 각 레벨별로 **무엇을 만들지**, **어떤 순서로 작업할지**, **어떤 도구와 데이터가 필요한지**만 정리했습니다.

---

## 1. 온톨로지 기본 개념 요약

| 개념 | 설명 |
|------|------|
| **온톨로지** | 특정 도메인의 개념(클래스), 관계(속성), 제약을 명시적으로 정의한 모델. 지식의 공유·재사용·추론의 기반이 됨. |
| **클래스(Class)** | 동일한 성질을 가진 개체들의 집합 (예: Person, Product, Event). |
| **인스턴스(Instance)** | 클래스에 속하는 구체적인 개체 (예: "홍길동"은 Person의 인스턴스). |
| **속성(Property)** | 개체 간 관계 또는 개체의 속성 (예: hasParent, createdAt). |
| **계층(Hierarchy)** | 클래스/속성의 is-a 관계로 이루어진 트리 (예: Dog ⊑ Animal). |

이 예제는 위 개념을 **초급 → 중급 → 고급** 순으로 다룹니다.  
**초급**은 원래 예제(PubMed 논문 메타데이터)대로 진행하고, **중급부터**는 "근거 중심 지식 그래프" 도메인을 적용합니다 (각 레벨 README 참고).

---

## 2. 레벨별로 만들 것

### 초급 (01-beginner)
- **목표**: 온톨로지가 뭔지, 클래스/인스턴스/속성이 어떻게 정의되는지 이해.
- **만들 것**:
  - 단순 도메인(예: 사람, 조직, 역할)에 대한 **소규모 온톨로지** (클래스 5~10개, 속성 몇 개).
  - **샘플 데이터**: CSV/JSON 형태의 인스턴스 몇 개.
  - 온톨로지를 **파일로 저장** (RDF/Turtle 또는 OWL)하고, **로드·검증**하는 최소 스크립트.
- **결과**: “개념 정의 + 데이터 로드 + 검증” 한 사이클을 경험.

### 중급 (02-intermediate)
- **도메인**: 근거 중심 지식 그래프 (문헌·생물의학 개체·MENTIONS/ASSOCIATED_WITH). `02-intermediate/README.md` 참고.
- **목표**: 온톨로지 확장, 관계 쿼리, 그래프/관계형 DB 활용.
- **만들 것**:
  - 초급에서 익힌 온톨로지를 **확장**해 위 도메인 반영 (클래스·속성 추가, 도메인/범위 제약).
  - **Neo4j 또는 PostgreSQL**에 온톨로지·데이터를 반영한 스키마/그래프 모델.
  - **변환 스크립트**: RDF/OWL → Neo4j 또는 PostgreSQL.
  - **예제 쿼리**: 특정 관계 경로 조회, 집계 등 (Cypher 또는 SQL).
- **결과**: “온톨로지 → 저장소 → 쿼리” 파이프라인 구축.

### 고급 (03-advanced)
- **목표**: 다중 온톨로지 통합, 추론, API/서비스화.
- **만들 것**:
  - **여러 도메인 온톨로지** 통합 (매핑, 상속, equivalences). **(선택) PrimeKG** 같은 통합 지식 그래프와 노드/ID 매핑으로 drug·disease·gene 데이터 연계.
  - **추론 규칙** 적용 (예: transitive 관계, 분류 추론) — 선택 시 SPARQL 또는 규칙 엔진.
  - **ETL 파이프라인**: 원본 데이터 → 온톨로지 기반 모델 → Neo4j/PostgreSQL.
  - **REST API** 또는 **SPARQL 엔드포인트**로 쿼리 노출 (선택).
- **결과**: “통합 온톨로지 + 추론 + 운영 가능한 데이터 서비스”까지 경험.

## 4. 사용할 도구 (Tools)

| 용도 | 도구 | 비고 |
|------|------|------|
| **온톨로지 편집/시각화** | Protégé, TopBraid, VocBench | OWL/RDF 편집, 계층 확인. |
| **형식** | RDF(Turtle `.ttl`), OWL (`.owl`) | 초급은 Turtle만으로도 가능. |
| **저장소 (그래프)** | **Neo4j** | 노드–관계 모델, Cypher 쿼리. 온톨로지의 클래스=레이블, 속성=관계로 매핑. |
| **저장소 (관계형)** | **PostgreSQL** | 테이블=클래스, FK=관계. JSONB로 속성 저장 가능. |
| **RDF 저장소 / SPARQL** | Apache Jena Fuseki, GraphDB, Virtuoso | 고급에서 SPARQL·추론 사용 시. |
| **스크립트 언어** | Python (rdflib, py2neo, psycopg2), Node.js | 변환·로드·쿼리 자동화. |

- **초급**: Protégé(또는 텍스트 에디터) + Python rdflib 정도면 충분.  
- **중급**: Neo4j **또는** PostgreSQL 중 하나 선택.  
- **고급**: Neo4j/PostgreSQL + (선택) Jena/GraphDB로 SPARQL·추론.

## 5. 필요한 데이터 (Data)

### 초급
- **데이터 종류**: 도메인에 맞는 **소량 인스턴스** (예: 사람 5~10명, 조직 2~3개, 역할 몇 개).
- **형식**: CSV 또는 JSON.
- **내용**: 각 개체의 ID, 이름, 타입(클래스), 관계(예: 소속 조직, 역할).
- **출처**: 직접 작성해도 됨. 공개 데이터셋(예: DBpedia 일부)을 잘라 써도 됨.

### 중급
- **데이터 종류**: 초급보다 **량·관계 수**를 늘린 데이터 (인스턴스 수십~백 단위, 관계 다양).
- **형식**: CSV/JSON 또는 RDF(Turtle).
- **조건**: 온톨로지의 클래스·속성과 1:1로 매핑 가능해야 함.

### 고급
- **데이터 종류**: **여러 출처**의 데이터 (예: 인물 DB + 제품 카탈로그).
- **형식**: CSV, JSON, RDF 혼합 가능.
- **조건**: 서로 다른 스키마를 온톨로지로 **매핑**할 수 있어야 함. ID 연결용 키(또는 URI) 필요.

## 6. PubMed 데이터로 테스트하기

**세 레벨 모두 PubMed 논문 데이터로 테스트할 수 있습니다.**  
PubMed는 논문(Article), 저자(Author), 저널(Journal), MeSH 용어 등이 명확해 온톨로지 학습에 적합합니다.

### 레벨별 적합성

| 레벨 | PubMed로 테스트 가능? | 설명 |
|------|------------------------|------|
| **초급** | ✅ 적합 | 논문·저자·저널·MeSH 등 클래스 정의 → 소량 논문만으로 구조 이해. |
| **중급** | ✅ 적합 | 확장 온톨로지 + Neo4j/PostgreSQL 로드 → "저자별 논문 수", "MeSH별 논문" 등 쿼리. |
| **고급** | ✅ 적합 | MeSH 계층(또는 다른 온톨로지)과 논문 데이터 통합, 추론(하위 MeSH 포함 검색), ETL/API. |

### 최소 논문 건수 (권장)

| 레벨 | 최소 논문 수 | 권장 범위 | 이유 |
|------|--------------|-----------|------|
| **초급** | **10건** | 10~30건 | 클래스·인스턴스·속성만 확인하면 되므로 소량으로 충분. |
| **중급** | **50건** | 50~200건 | 저자/저널/MeSH 분포가 나와야 집계·관계 쿼리가 의미 있음. |
| **고급** | **100건** | 100~500건+ | 통합·추론·ETL을 보여주려면 어느 정도 규모 필요. |

- **데이터 출처**: [PubMed E-utilities](https://www.ncbi.nlm.nih.gov/books/NBK25501/) (API), 또는 XML/JSON 다운로드.  
- **초급만** 빠르게 돌려보려면 **최소 10건**이면 됩니다. 중급·고급은 위 권장 건수를 참고해 조절하면 됩니다.

### PubMed E-utilities: PMID vs PMC, 메타데이터 vs 원문

- **BASE_URL** (`eutils.ncbi.nlm.nih.gov/entrez/eutils`)는 공통 진입점이며, **db** 파라미터로 DB가 정해집니다.
- **db=pubmed** → **PubMed** 조회. 반환 ID는 **PMID**(PubMed ID). **PMC ID가 아님.** efetch 시 **메타데이터+초록**만 제공하며 **본문 원문은 없음.**
- **db=pmc** → **PMC**(PubMed Central) 조회. 반환 ID는 **PMC ID**. 오픈액세스 논문은 **전체 원문** XML 제공 가능.

**정리**: 현재 사용 중인 `db=pubmed`는 PMID 기준으로 **제목·초록·저자·저널·MeSH**만 가져옵니다. 논문 전체 원문이 필요하면 **db=pmc**로 PMC ID를 조회한 뒤 efetch해야 합니다.

### 온톨로지 구축 시 정확히 필요한 데이터

이 예제(초/중/고급)의 목표는 개념(클래스)·관계(속성)·인스턴스를 정의하고 쿼리하는 것이므로 **논문 전체 원문은 필요 없습니다.**

| 필요한 것 | PubMed(db=pubmed)로 충분? |
|-----------|---------------------------|
| 클래스 정의(Article, Author, Journal, MeSH 등) | 정의만 하면 됨. |
| 인스턴스 식별자(PMID, 저자명, 저널명, MeSH) | ✅ 메타데이터에 모두 있음. |
| 관계(논문–저자, 논문–MeSH 등) | ✅ 메타데이터만으로 구성 가능. |
| 초록(검색/필터용) | ✅ efetch에 포함. |
| 논문 본문(원문) | ❌ db=pubmed에는 없음. 본문에서 개체/관계 추출할 때만 db=pmc 원문 필요. |

**결론**: 초급·중급·고급 모두 **메타데이터(PMID, 제목, 초록, 저자, 저널, MeSH)** 만 있으면 됩니다. 전체 원문은 필요 없고, 지금처럼 `db=pubmed`로 가져온 데이터로 충분합니다.

### 초급: 데이터는 API만 가져오면 되나? Cleansing / Chunking 필요?

| 항목 | 초급에서 필요한가? | 설명 |
|------|-------------------|------|
| **API로 가져오기** | ✅ **핵심** | PubMed E-utilities로 논문 메타데이터(PMID, 제목, 저자, 저널, MeSH 등)를 받으면 됨. |
| **Cleansing** | ⚠️ **가벼운 정도만** | 빈 값 처리, 앞뒤 공백 제거, 인코딩 정리 정도. API 응답이 이미 구조화되어 있어 본격 정제는 불필요. |
| **Chunking** | ❌ **필요 없음** | 초급에서는 **메타데이터**로 클래스·인스턴스·관계만 다룸. 초록을 잘게 쪼개는 건 임베딩/RAG용이라 온톨로지 기본 예제와 별개. |

**정리**: 초급은 **API 호출 → 응답(JSON/XML)을 그대로 또는 약간만 정리 → CSV/JSON으로 저장**하면 충분. Chunking은 하지 않아도 됨.

### 중급: Cleansing / Chunking 필요?

| 항목 | 중급에서 필요한가? | 설명 |
|------|-------------------|------|
| **API/추출 데이터** | ✅ **핵심** | PubMed 메타데이터 또는 추출기 출력(문서·개체·MENTIONS/ASSOCIATED_WITH)을 JSON 등으로 준비. |
| **Cleansing** | ⚠️ **가벼운 정도** | 초급과 동일. 빈 값·공백·인코딩 정리 정도. 본격 정제는 불필요. |
| **Chunking** | ❌ **RAG용 세밀 chunking 불필요** | 중급 온톨로지에는 **EvidenceChunk**(abstract/섹션 단위)가 있지만, 이는 “근거 단위” 구조 개념. **초록 전체를 1청크**로 두거나 **문서당 1청크**로 두면 됨. 문장/토큰 단위로 잘게 쪼개는 건 임베딩·RAG용이라 “온톨로지 → 저장소 → 쿼리” 목표에는 필요 없음. |

**정리**: 중급도 **메타데이터(또는 시뮬레이션 JSON) → EvidenceChunk는 문서당 1개(또는 초록=1청크)로 구성 → 변환·로드** 하면 됨. RAG용 Chunking은 하지 않아도 됨.

```

│   └── scripts/              ← 로드·검증

```
